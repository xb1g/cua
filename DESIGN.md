## Problem Statement

Northstar (Tzafon's CUA model) has weak per-step accuracy. Naive single-pass
agent runs fail too often to be useful. We want a hackathon-winnable submission
that turns a mediocre CUA model into a reliable scraper for arbitrary URLs by
exploiting one fact: **scraping has cheap verification signals** (row counts,
schema match, type validity, dedup, "did we paginate?"). Verification = reward,
which means we can both *loop until success* AND *train a critic on the
trajectories we generated*.

Target: "point at any URL, describe a table, get the table." Heterogeneous and
ambitious — counterbalanced by a hard time-box and a strict-subset fallback.

## What Makes This Cool

- The agent **fails on stage and self-corrects on stage.** Judges see the loop
  visibly recover from a mistake. That's a stronger demo than any single hero
  example.
- Open-source artifact: an **MIT-licensed loop wrapper** that works against
  Northstar, Anthropic CUA, or any vision model with `screenshot → action`
  semantics. Direct contribution to the open-source CUA stack.
- Online RL critic trained DURING the hackathon on trajectories generated by
  your own agent — not pre-baked. That story is rare at hackathons.
- Concrete novelty narrative: "first 50 attempts are pure loop. Last 10 use the
  reranker we trained at lunch on Brev. 2x success rate."

## Premises (agreed)

1. Northstar's per-step accuracy is the bottleneck — not vision, not planning.
2. Scraping has cheap verifiers (row count, schema match, type validity, dedup,
   pagination check) — so we get reward signal for free.
3. 6 hours is too short for fine-tuning the CUA itself. A small *critic* /
   reranker is feasible. Direct LoRA on the VLM is too risky.
4. An open-source artifact (MIT-licensed loop wrapper) beats a one-off
   benchmark number for this judge panel.

## Recommended Approach: Loop-first, RL-critic-second

### Phase 0 — Scaffold (30 min)
- Repo: `cua-verified-loop` (MIT). Single `bun` or `uv` project.
- Adapter interface: `CUAClient.step(screenshot, instruction) -> action`.
- Implement adapters: `NorthstarClient`, `AnthropicClient` (fallback / comparison).
- KERNEL session wrapper: spin up browser, take screenshots, dispatch actions.
- Trajectory logger: append (timestamp, screenshot_hash, action, page_state,
  verifier_signal) to JSONL per attempt.

### Phase 1 — The Loop (90 min, target done by 1:00pm)
- **Outer loop**: N parallel attempts on the same task. N starts at 5.
- **Inner step**: standard CUA loop (screenshot → model → action → screenshot).
- **Verifier (LLM-as-judge)**: cheap Sonnet/Haiku call after agent reports
  "done" — answers `{success: bool, rows_extracted: int, schema_valid: bool,
  reason: string}`.
- **Self-critique on failure**: feed the failed trajectory + verifier reason
  back as context for the next attempt. This alone often doubles success rate.
- **Stop conditions**: any attempt verified-success OR all N failed.
- Demo target by 1pm: works on 3 hand-picked URLs (one easy DOM site, one
  Tableau-style canvas, one login-walled flow with KERNEL).

### Phase 2 — Critic Training (90 min, lunch onwards on Brev)
- During Phase 1, every (state, action, eventual_outcome) tuple is logged.
- After Phase 1: filter trajectories into success/failure pairs.
- Train a tiny critic model: input = (screenshot embedding, candidate action),
  output = predicted success prob. Use a frozen CLIP/SigLIP image encoder + a
  small MLP head. ~few hundred to a few thousand pairs is enough to show signal.
- Brev GPU runs the training; main machine keeps generating data.

### Phase 3 — Inference-time Reranking (45 min)
- At each step, sample K=5 candidate actions from Northstar (temperature > 0).
- Critic scores each. Pick argmax.
- Compare: pure loop success rate vs loop + critic. **This is the headline
  metric for the demo.**

### Phase 4 — Demo polish (60 min, 3:00–4:00pm)
- Live web UI: shows current screenshot, action, verifier feedback, success
  rate counter for "loop only" vs "loop + critic". Two columns side-by-side.
- README with: 60-second loom, install in `npx cua-verified-loop init`, the
  metric chart, MIT license, link to Northstar.
- Submission: GitHub repo + 90-second video.

### Strict-subset fallback
If critic training fails by 2:30pm, drop Phase 2/3, polish Phase 1 harder.
You still have a working open-source self-correcting loop — that alone is a
respectable hackathon submission.

## Approaches Considered

### Approach A: Pure loop (oh-my-openagent style)
- **Pros**: ships fastest, bulletproof demo, clean open-source artifact.
- **Cons**: moderate technical novelty. RL story is missing.
- **Effort**: S. **Risk**: Low.

### Approach B: Loop + online critic (CHOSEN)
- **Pros**: best demo narrative ("trained at lunch"), fits dual-track judges
  (applications + research), A is strict subset → graceful degrade.
- **Cons**: critic training may not produce visible delta in time.
- **Effort**: M. **Risk**: Medium.

### Approach C: LoRA fine-tune Northstar / a small VLM
- **Pros**: maximum technical novelty.
- **Cons**: cliff failure mode — high probability of nothing working at 4pm.
- **Effort**: L. **Risk**: High. **Rejected.**

## Open Questions

- Does Tzafon expose Northstar logits / multiple-sample API? If only single
  greedy outputs are available, K-sample reranking degrades to "rerun with
  different temperature" — still works but less clean.
- KERNEL session quota — confirm cost per parallel browser early.
- Verifier prompt design: Sonnet is more accurate, Haiku is 5x cheaper — start
  with Haiku, escalate only if false-success rate is bad.
